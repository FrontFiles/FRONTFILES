E6.D is UI polish only. No reducer changes, no schema, no API, no engine. The data already flows through the widened proposal shape (per E6.B); E6.D is the consumer-side update that makes that data visible.

## 1. What E6.D is

A targeted edit to `src/app/vault/upload/_components/inspector/InspectorAIProposalDetail.tsx` that:

1. **Replaces the single overall-confidence reads** at the section header (line 78: `{overallConfPct}%`) and inside the "Tag confidence" body (line 117-118: `Overall metadata confidence: {overallConfPct}%`) with **per-field confidence reads** sourced from `proposal.description_confidence`, `proposal.keywords_confidence`, `proposal.tags_confidence`.
2. **Preserves the existing 3-row collapsible structure** (Caption rationale / Price basis / Tag confidence) — only the rendered values inside two of those rows change.
3. **Brings the inspector into compliance** with the spirit of E6 §17 don't-do #7 ("new code reads per-field directly") — the inspector is currently the only V4 component still reading the deprecated MAX-of-fields `proposal.confidence`.

E6.D ships dormant in production until `FFF_AI_REAL_PIPELINE=true` flips, at which point the per-field confidences become non-mock values from the real pipeline.

## 2. Authority + UX-SPEC-V4 §11 mapping

UX-SPEC-V4 §11 governs AI proposal surfacing visuals. The relevant treatment for the inspector detail panel:

| Field | Spec source | Visual treatment |
|---|---|---|
| Caption rationale | UX-SPEC-V4 §11.2 + L7 (collapsible per-row) | Existing — no change in E6.D |
| **Caption confidence** | UX-SPEC-V4 §11.1 (per-field auto-accept threshold) | **NEW in E6.D** — render `description_confidence` as percentage in the section header replacing `overallConfPct`, AND as a row in the "Tag confidence" body (which becomes "Confidence" — see §3 below for the rename) |
| **Keywords confidence** | UX-SPEC-V4 §11.1 | **NEW in E6.D** — render `keywords_confidence` as a percentage row |
| **Tags confidence** | UX-SPEC-V4 §11.1 | **NEW in E6.D** — render `tags_confidence` as a percentage row (replacing the current single overall-confidence read) |
| Price basis | UX-SPEC-V4 §11.3 | Existing — no change in E6.D |
| Tag chips | UX-SPEC-V4 §11.4 | Existing — no change in E6.D (the tag chips inside the "Tag confidence" row stay as-is; only the headline overall-confidence read changes) |

**Section header overall confidence:** the section header currently shows a single `{overallConfPct}%` (computed as `Math.round(asset.proposal.confidence * 100)`). Per E6 §17 #7, new code should read per-field directly. Two options:

- **(α)** Drop the header percentage entirely (inspector relies on per-field rows for confidence; header just shows the section title)
- **(β)** Keep a header percentage but compute as MAX-of-per-field at the component level (mirrors the hydration-side MAX-of-fields backward-compat aggregation per E6 §17 #7), explicitly reading per-field

Recommend **(β)** — a single MAX-of-fields header indicator preserves the at-a-glance affordance creators are accustomed to from the existing UI, while the body rows show the per-field breakdown when the section is expanded. The header MAX is computed inline (no reducer/state involvement) so the don't-do #7 spirit is honored.

## 3. Testable exit criteria (per AGENTS.md gate)

E6.D is complete when ALL the following are true:

1. **Section header**: shows `Math.max(description_confidence ?? 0, keywords_confidence ?? 0, tags_confidence ?? 0)` rendered as `{percent}%` — no read of `proposal.confidence`.
2. **"Tag confidence" row, renamed to "Confidence"**: body renders THREE rows (description / keywords / tags), each showing `{label}: {percent}%`. Per-field values come from `proposal.description_confidence`, `proposal.keywords_confidence`, `proposal.tags_confidence`. Tag chips stay below the three confidence rows (existing behavior).
3. **Fallback for missing per-field**: any per-field value that is `undefined` or `null` renders as `—` (em-dash) rather than `0%`, distinguishing "no data" from "zero confidence." Pre-E3 simulation fixtures may lack these fields per `v2-types.ts:160-166` (marked optional); the component must handle that.
4. **No new state**: no React state additions; no reducer dispatches; no context changes. Pure render-layer change.
5. **No regressions in 76 V3 reducer parity tests**: parity tests don't exercise this component directly, but `npx vitest run src/lib/upload` MUST stay green.
6. **Existing snapshot tests** for `InspectorAIProposalDetail` (if any) update with the new render shape; visual diff bounded to the changed rows.
7. **`npx tsc --noEmit` green** — same baseline as post-`chore/tsc-baseline-cleanup-v2`.
8. **BP-D7 copy review**: the three new row labels ("Description confidence", "Keywords confidence", "Tags confidence") follow sentence case + advisory framing per the language discipline already established in E6 §13.

## 5. Implementation outline

### 5.1 Replace section-header overall confidence

Current (lines ~63-78):

```typescript
const overallConfPct = Math.round(asset.proposal.confidence * 100)
// ... rendered as
<span className="text-slate-500 font-mono">{overallConfPct}%</span>
```

After (per §3 exit criterion 1):

```typescript
const desc = asset.proposal.description_confidence ?? 0
const kw = asset.proposal.keywords_confidence ?? 0
const tg = asset.proposal.tags_confidence ?? 0
const headerMaxPct = Math.round(Math.max(desc, kw, tg) * 100)
// ... rendered as
<span className="text-slate-500 font-mono">{headerMaxPct}%</span>
```

### 5.2 Replace "Tag confidence" row body with three per-field rows

Current (lines ~110-135):

```tsx
<DetailRow open={tagsOpen} onToggle={() => setTagsOpen(o => !o)} label="Tag confidence">
  <div className="flex flex-col gap-1 text-xs">
    <div className="text-slate-700">
      Overall metadata confidence:{' '}
      <span className="font-mono text-black">{overallConfPct}%</span>
    </div>
    {asset.proposal.tags.length > 0 ? (
      <div className="flex flex-wrap gap-1">
        {asset.proposal.tags.map((t, i) => (
          <span key={`${t}-${i}`} className="...">{t}</span>
        ))}
      </div>
    ) : (
      <span className="italic text-slate-500">No tags suggested.</span>
    )}
  </div>
</DetailRow>
```

After:

```tsx
<DetailRow open={tagsOpen} onToggle={() => setTagsOpen(o => !o)} label="Confidence">
  <div className="flex flex-col gap-1 text-xs">
    <ConfidenceRow label="Description" value={asset.proposal.description_confidence} />
    <ConfidenceRow label="Keywords" value={asset.proposal.keywords_confidence} />
    <ConfidenceRow label="Tags" value={asset.proposal.tags_confidence} />
    {asset.proposal.tags.length > 0 ? (
      <div className="flex flex-wrap gap-1 mt-2">
        {asset.proposal.tags.map((t, i) => (
          <span key={`${t}-${i}`} className="...">{t}</span>
        ))}
      </div>
    ) : (
      <span className="italic text-slate-500">No tags suggested.</span>
    )}
  </div>
</DetailRow>
```

`ConfidenceRow` is a small inline component (defined in the same file, like `DetailRow`):

```tsx
function ConfidenceRow({ label, value }: { label: string; value: number | null | undefined }) {
  const display = value == null ? '—' : `${Math.round(value * 100)}%`
  return (
    <div className="text-slate-700 flex justify-between">
      <span>{label}</span>
      <span className="font-mono text-black">{display}</span>
    </div>
  )
}
```

### 5.3 Row label rename

The collapsible row's label changes from "Tag confidence" to "Confidence" (now covers all three fields, not just tags). This is a small string change; the existing `DetailRow` component takes the label as a prop.

### 5.4 No reducer / no state

`overallConfPct` is removed entirely from the component. The header value (`headerMaxPct`) is computed inline at render time. No new `useState` calls. No dispatches.

## 9. Don't-do list

1. **Don't change the reducer or any state machine.** E6.D is render-layer only. Any reducer touch is out of scope and would re-open the 76 V3 parity test surface.
2. **Don't compute or stash `overallConfPct` derived state in context or reducer.** The MAX-of-fields header value is computed inline at render. No context state.
3. **Don't read `asset.proposal.confidence`.** Per E6 §17 #7. Use the per-field fields directly. The deprecated `confidence` field stays available on the type for backward-compat with consumers that haven't migrated, but the inspector migrates here.
4. **Don't redesign the collapsible structure.** Three-row layout (Caption rationale / Price basis / Confidence) stays. Only the body of the renamed row changes.
5. **Don't add new UI surfaces.** No tooltips, no popovers, no new affordances. Three rows of `{label}: {percent}%` plus existing tag chips.
6. **Don't render `0%` when the field is missing.** Distinguish missing (`—`) from explicit zero (`0%`). Pre-E3 simulation fixtures lack the widened fields per `v2-types.ts:160-166`.
7. **Don't change BP-D7 copy posture.** Row labels are sentence case; no certifying language; no "AI-verified" framing.
8. **Don't extend AIProposalBanner.tsx.** Per audit, the banner renders `proposal.rationale` directly; no per-field confidence display is needed there. AIProposalBanner is a separate concern from per-asset confidence display.
9. **Don't ship before `FFF_AI_REAL_PIPELINE` flag flip planning.** E6.D ships behind the same flag as the rest of E6. Visual change is dormant in production until the flag flips, at which point the simulated MAX-of-fields → real per-field values transition is automatic (no separate cutover step for E6.D).
10. **Don't compose F-track price-basis changes here.** The "Price basis" row stays as-is; price-engine integration is L6 + F-track scope.
