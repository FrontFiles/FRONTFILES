# D2.4 — Right Rail Inspector (Implementation Directive)

**Status:** DRAFT — pending founder ratification before implementation
**Date:** 2026-04-26
**Branch target:** `feat/d-upload-v4` (continues from `fdda0da`)
**Governs:** the D2.4 sub-prompt — the persistent right rail that hosts per-asset editing, AI proposal review, exceptions, duplicate resolution, and the "Set as cover" affordance. The only place per-field AI furniture exists.
**Reads underlying:** `docs/upload/UX-SPEC-V4.md` §5 (canonical) + §7 (cover) + §11 (AI surfacing), `docs/upload/D-PLAN.md` §10 (scope), the D2.1 spine (`getStoryCover`, `SET_STORY_COVER`), C2 spine survivors that adapt forward (`DuplicateResolver`, `ConflictResolver`, `PriceBasisPanel`)

## 0. Context

D2.1 stood up the shell with a placeholder right rail (`Right rail — placeholder for D2.4` mounting on `selectedAssetIds.length === 1`). D2.3 made the contact sheet selectable. Click → placeholder mounts. D2.4 fills it.

This is **the only surface where per-field AI furniture lives.** Per UX-SPEC-V4 §11.3: "No per-field accept furniture on the contact sheet" — the inspector is where ✓ + ↻ + Why? affordances exist for every editable field that has a non-accepted AI suggestion.

D2.4 also lands the **"Set as cover" button** as the second co-equal primary path for setting a story cover (per UX-SPEC-V4 §7.3 — the first path is drag-asset-onto-story-header in the left rail, which lands at D2.2).

Most of the content adapts forward from the C2 SideDetailPanel (now dormant). Same field editor, same exceptions section, same AI Proposal Detail body. Three changes:

1. **Mount mechanism**: persistent right rail when `selectedAssetIds.length === 1`, not push-from-right slide-in
2. **Thumbnail**: 16:9 landscape per the founder lock; ~400×225 in the 400px-wide rail
3. **"Set as cover" button** added when the open asset is in a story

## 4. Implementation Points (ratify before implementation)

### IPD4-1 — Component reuse strategy
The C2 SideDetailPanel content (FieldEditor, ExceptionsSection, AcceptRow, AIProposalDetailSection) lives inside the now-dormant SideDetailPanel.tsx. How does D2.4 reuse?

- **(a)** **Copy + adapt into a new `_components_v4/inspector/` subdirectory.** Each helper becomes its own file. D2.8 cutover deletes the dormant SideDetailPanel along with its embedded copies. Recommended — no cross-import from dormant code; each version owns its lifecycle.
- (b) Extract the helpers from the dormant SideDetailPanel into a shared neutral location both can import.
- (c) Import from the dormant file directly.
- **Default: (a).** Dormant code is dormant — should not be a live source of truth.

### IPD4-2 — Persistent vs conditional mount
- **(a)** Mount only when `selectedAssetIds.length === 1` (per IPV4-1 default = a). Recommended.
- (b) Add a "pin" toggle to keep the rail mounted with empty state.
- **Default: (a).**

### IPD4-3 — Progressive disclosure default state
Per L7, six fields are always visible and the rest are collapsible. Default state of the collapsible sections:
- **(a)** All collapsed by default — clean inspector load. User expands what they need. Recommended.
- (b) All expanded by default — no hidden content.
- (c) Smart default — Exceptions section auto-expanded if any blocking exception present; rest collapsed.
- **Default: (a).** (c) is a viable enhancement post-D2.4 if user testing shows it helps.

### IPD4-4 — "Set as cover" button visibility
- **(a)** Visible when `asset.storyGroupId !== null`. When `state.storyGroupsById[asset.storyGroupId].coverAssetId === asset.id`, button reads `✓ Cover` and is disabled. Recommended.
- (b) Always visible (disabled when no story).
- (c) Hidden when asset is already cover (no disabled state).
- **Default: (a).**

### IPD4-5 — Thumbnail dimensions in the 400px rail
Per L6 (16:9) + spec §5.2 (up to 320px tall). 400 × (9/16) = 225px tall.
- **(a)** Full-width 16:9 = 400 × 225. Recommended.
- (b) 320px tall as ceiling — width derived = 320 × (16/9) = 569px (overflows the 400px rail). Reject.
- (c) Use `aspect-video` Tailwind class with `w-full`. Cleanest CSS expression of (a).
- **Default: (a) implemented as (c).**

### IPD4-6 — Keyboard nav action choice
Reuses `NAVIGATE_SIDE_PANEL` action from D2.1 spine? The action's name is C2-era (sidePanelOpenAssetId-themed) but its handler also updates `selectedAssetIds`.
- **(a)** Reuse `NAVIGATE_SIDE_PANEL` as-is. The action also writes selectedAssetIds, which is what D2.4 reads. The vestigial sidePanelOpenAssetId write is harmless. Recommended.
- (b) Add a new `NAVIGATE_SELECTION` action; deprecate `NAVIGATE_SIDE_PANEL` at D2.8 cutover.
- (c) Add a new action; remove `NAVIGATE_SIDE_PANEL` now.
- **Default: (a).** No naming purity is worth the additional reducer churn.

### IPD4-7 — Esc behavior
- **(a)** Dispatches `DESELECT_ALL_ASSETS` — closes the rail by clearing selection. Recommended; matches the mount-on-selection model.
- (b) Dispatches a no-op (rail can only be closed by clicking outside).
- **Default: (a).**

### IPD4-8 — Click-outside-to-close mechanism
The rail mounts on selection. To close it, user has to deselect. Mechanisms:
- (a) Click empty area of contact sheet → triggers `DESELECT_ALL_ASSETS`. Requires CenterPane to have a click-handler. Already implicit-ish (contact sheet card click replaces selection; clicking gap doesn't fire anything currently).
- **(b)** Click another card → selection moves; rail re-renders with new asset. Click "All" filter chip to clear search shouldn't deselect. Esc deselects (per IPD4-7). No explicit click-outside handler — keeping it simple. Recommended.
- (c) Wire a click-outside handler on the rail itself (closes when click lands outside).
- **Default: (b).** Esc + click-another-card cover the common cases. Click-on-empty-canvas can be added later if user testing demands.

### IPD4-9 — Empty rail state (when triggered to open but asset is gone)
What happens if `selectedAssetIds.length === 1` but the id no longer exists in `assetsById` (e.g., excluded → hidden from view) — same defensive question as IPIII-5 in C2.3.
- **(a)** Rail self-deselects via `useEffect` watching `assetsById[selectedId]`. Same pattern as C2 SideDetailPanel did. Recommended.
- (b) Rail renders an "asset not found" placeholder.
- **Default: (a).**

### IPD4-10 — InspectorAcceptRow extraction
- **(a)** Extract as its own file in `_components_v4/inspector/InspectorAcceptRow.tsx`. Recommended for clarity (single-responsibility file).
- (b) Keep inline inside InspectorFieldEditor.
- **Default: (a).**

### IPD4-11 — Test depth
- **(a)** No new reducer or selector tests. Visual smoke is the gate. Same posture as D2.3.
- (b) Extract a tiny unit test for the cover-resolution selector behavior under "Set as cover" flow.
- **Default: (a).** `getStoryCover` already has selector tests from D2.1.

### IPD4-12 — Inspector visibility when commit phase != idle
During `committing` / `success` / `partial-failure` phases, should the inspector still mount?
- **(a)** No — when `state.commit.phase !== 'idle' && phase !== 'summary'`, suppress the rail. Editing during commit is meaningless. Recommended.
- (b) Yes — always mount on selection regardless of commit state.
- **Default: (a).** Matches the C2 commit-phase model where editing affordances quiet down during commit.

## 5. Acceptance criteria

- Single-click any contact-sheet card → right rail mounts with that asset's editor on the right side
- Right rail width: 400px fixed
- 16:9 landscape thumbnail at top, full rail width (≈400×225)
- Always-visible fields: title (input), caption (textarea), price (number), privacy (select), Set-as-cover button (when applicable)
- Collapsible sections (closed by default): tags, geography, licences, exceptions, AI Proposal Detail
- Per-field ✓ + ↻ icons appear next to fields with non-accepted AI suggestions (proposal.confidence < 0.85 → ghost remains; per-field accept lives here)
- "Set as cover" button: visible when asset has `storyGroupId`; reads "✓ Cover" disabled when already cover; click dispatches `SET_STORY_COVER`
- DuplicateResolver embedded in exceptions section when asset has `duplicateStatus === 'likely_duplicate'`
- ConflictResolver embedded for unresolved metadata conflicts; dispatches `RESOLVE_CONFLICT`
- PriceBasisPanel renders inside AI Proposal Detail "Price basis" expand row
- Esc → deselects (rail closes)
- J / K → navigates to next/prev visible asset (selection follows; rail re-renders)
- Click another card → rail updates to that asset
- Multi-select (≥2) → rail unmounts (D2.5 contextual action bar will fill the gap; for now nothing replaces it)
- Commit phase != idle/summary → rail suppressed
- All field edits dispatch `UPDATE_ASSET_FIELD`
- All 172 existing tests still pass (regression gate)
- Bulk-accept-price guard remains intact (no UI affordance for bulk price; per-asset ✓ on price field is single dispatch)
