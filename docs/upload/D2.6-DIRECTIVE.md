## §1 What this is

Compare mode is the focused 2-up review: two compared assets shown side-by-side at large preview size so the creator can decide between them (e.g., choose between two near-duplicates, or evaluate two takes of the same shot).

It's invoked from the contextual action bar (D2.5) when the creator has exactly 2 assets selected and clicks **Compare**. The bar dispatches `ENTER_COMPARE_MODE { assetIds }`, which sets `state.ui.compareAssetIds`. `getLayoutState` returns `'comparing'`, and `CenterPane` swaps from the contact sheet to a `CompareView` 2-up grid.

Per the founding lock + IPV4-3 default = (a): **strict 2-only**. 3+ selected disables the Compare button (this is already enforced in D2.5; no change needed).

## §5 CompareView anatomy

### §5.1 Layout

```
┌──────────────────────────────────────────────────────────────────┐
│  COMPARE                                              EXIT (Esc) │   ← header bar (h-10)
├──────────────────────────────┬───────────────────────────────────┤
│                              │                                   │
│      ┌────────────────┐      │      ┌────────────────┐           │
│      │                │      │      │                │           │
│      │   16:9 LARGE   │      │      │   16:9 LARGE   │           │
│      │   PREVIEW      │      │      │   PREVIEW      │           │
│      │                │      │      │                │           │
│      └────────────────┘      │      └────────────────┘           │
│                              │                                   │
│      filename_a.jpg          │      filename_b.jpg               │
│                              │                                   │
└──────────────────────────────┴───────────────────────────────────┘
```

- **Header bar (top, h-10):** "COMPARE" label left + filename count or asset names; "EXIT (Esc)" button right
- **Two columns:** 50/50 split via `grid grid-cols-2 divide-x divide-black flex-1 min-h-0`
- **Each column** (`CompareCard`):
  - 16:9 preview centered (object-contain, max-w/max-h within column bounds — spec §10.2 "fills its column")
  - Filename below preview, centered, mono small uppercase

### §5.2 Mount condition

`CompareView` is only mounted by `CenterPane` when `getLayoutState() === 'comparing'`, which requires `compareAssetIds.length === 2`. So `CompareView` can assume both ids exist — but defensively reads `state.assetsById[id]` and skips render with a warning if either is missing (defensive — same pattern as `RightRailInspector`'s stale-id guard).

### §5.3 Exit affordances

- **"EXIT (Esc)" button** in the header (always visible) → dispatches `EXIT_COMPARE_MODE`
- **Esc keypress** anywhere in the page → dispatches `EXIT_COMPARE_MODE`
  - Skip when typing inside an `INPUT/TEXTAREA/SELECT` (none in compare view by default, but defensive for the future)
- **Exit returns to Workspace layout** automatically via `getLayoutState` (compareAssetIds → [] → length !== 2 → fallthrough to 'workspace')
- **Selection persists** across exit (the 2 assets remain selected; the contextual action bar reappears)

## §7 ContextualActionBar interaction

**Problem:** `ENTER_COMPARE_MODE` does NOT clear `selectedAssetIds`. So when the user clicks Compare, selection remains at 2, and the contextual action bar's mount condition (`selectedCount >= 2`) keeps it visible. Without intervention, the bar would float at the bottom of the compare view — confusing and against L5 ("bulk-only-when-bulk").

**Fix:** Add ONE early-return to `ContextualActionBar.tsx`:

```tsx
// D2.6: bar is suppressed during compare mode (L5: bulk is contextual on
// multi-select in the workspace layout, not in compare). Selection persists
// across compare so the bar can reappear cleanly on exit.
if (state.ui.compareAssetIds.length === 2) return null
```

Placed immediately after the existing mount-condition checks (current lines 77-78). This is the smallest possible intervention — no reducer changes, no selection clearing.

## §8 Right-rail behavior in compare (IPD6-3)

**Decision required:** Should the right-rail inspector be available during compare mode?

The audit confirmed two facts:
- D-PLAN §12.1 says "Right rail still conditional"
- D-PLAN §12.2 says "Right rail can still open on either compared asset"

But the rail's mount condition is `selectedAssetIds.length === 1`. In compare, selection stays at 2 (the two compared assets). So under current wiring, the rail doesn't mount during compare. To "open the rail on a compared asset", the user would have to click that compared asset's header to single-select it, which would set `selectedAssetIds = [thatId]`, length === 1, rail mounts as a third column to the right of the compare view.

Three options:

- (a) **Suppress the rail entirely during compare.** Cleanest 2-up experience (full width for the two previews). To edit fields, user exits compare and selects the asset normally. Adds the same early-return pattern as the bar (§7).
- (b) **Allow the rail to mount during compare (current default).** User clicks a compare card header → selectedAssetIds = [thatId] → rail mounts as 3rd column. Visually busy (two large compare panels + 400px rail = ~1600px+ minimum width).
- (c) **Embed inspector inline in each compare card.** Most info-dense but most over-built. Spec §10.2 explicitly keeps compare minimal.

**Recommended default = (a).** Suppress the rail during compare. Matches spec's minimal compare anatomy. The user invokes compare to FOCUS on the two assets — the rail breaks that focus. To edit, exit (Esc) → rail comes back via normal selection.

## §9 Selection state in compare (IPD6-4)

When the user is in compare mode, what should clicking a compare card do?

Options:

- (a) **No-op.** Compare cards aren't clickable (read-only view). User can only Exit, then re-select normally.
- (b) **Single-select.** Click a card → selectedAssetIds = [thatId]. Combined with §8 = (b), this would mount the rail. With §8 = (a), this is a no-op visually but sets internal state.
- (c) **Toggle from compare set.** Click a card → that card is removed from compareAssetIds, EXIT_COMPARE_MODE fires (since length drops to 1).

**Recommended default = (a).** Compare cards are read-only. To do anything, exit first. Simplest mental model.

## §12 Risks

- **Header design under-specified.** Spec §10 doesn't specify the header bar pixel design. The default (h-10, COMPARE label left, EXIT button right) is a reasonable brutalist fit but may need polish later. Punt to post-ship feedback.
- **Preview size on narrow viewports.** At <800px center pane width, two 16:9 previews + dividers may cramp. The `max-w-[800px]` on each preview handles wide viewports; min content size (with `min-w-0`) keeps them from forcing horizontal scroll. On narrow displays they'll just be small — acceptable.
- **Object-URL leak in dev** — same trade-off as InspectorThumbnail (per IPD3-9). Production paths use real backend URLs; dev leaks are freed on page refresh.
