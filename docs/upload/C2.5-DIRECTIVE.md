## 0. Context

The reducer surface for AI proposals is **already in place** — eight actions handled with strict guards. Per IPI-1 + IP-5: `ACCEPT_PROPOSAL`, `BULK_ACCEPT_PROPOSALS_FOR_GROUP`, `BULK_ACCEPT_PROPOSALS_FOR_SELECTION` are **type-safe no-ops** in the reducer; actual acceptance is selector-derived from comparing `editable` vs `proposal`, and the auto-commit happens via `UPDATE_ASSET_FIELD` when the user edits (or accepts) the field. `REGENERATE_PROPOSAL` is a stub. `RECEIVE_AI_CLUSTER_PROPOSAL` / `ACCEPT_AI_CLUSTER_PROPOSAL` / `DISMISS_AI_CLUSTER_PROPOSAL` / `TOGGLE_PRICE_BASIS_PANEL` are all live.

Two C2.5-related surfaces have **already shipped** in earlier phases and just need verification + completion:
- `AIProposalBanner.tsx` — built in C2.2 §3.8, already dispatches accept/dismiss
- `AssetRow.tsx` (Linear mode) — already has `caption ✓` accept, `price ✓` accept, and a `Why?` button that toggles `priceBasisOpenAssetId`. But the panel that "Why?" should reveal **doesn't exist yet** (PriceBasisPanel.tsx is unbuilt).

C2.5 fills the remaining gaps: per-field acceptance UI in Linear and the side panel; a PriceBasisPanel component for the "Why?" expand and the side-panel detail; an AI Proposal Detail body in the side panel; bulk-accept wiring (no-op reducer + sequenced UPDATE_ASSET_FIELD client-side); regenerate UI; and a dev-mode mechanism to seed cluster proposals so the banner actually appears in fixtures.

This is the **final C2 structural sub-prompt.** After it lands, the C2.6 visual polish sweep can operate on a complete surface.

## 4. Implementation Points (ratify before implementation)

### IPV-1 — Per-field ✓ icons in AssetRowCompact?
Compact rows are intentionally tight (one-line summaries with chips on the right). Adding per-field ✓ would require expanding the row OR cramping the existing layout.

- **(a)** **No.** Compact rows stay summary-only. Per-field accept happens via (1) the side panel's per-field ✓ icons, or (2) the cluster header's "Accept all suggestions" bulk action. Recommended.
- (b) Yes — add a single ✓ that bulk-accepts caption+tags+keywords for that one row (no per-field granularity).
- (c) Yes — expand the compact row to two lines when proposals are present.
- **Default:** (a). Side panel + cluster bulk give two paths to accept; row-level ✓ is redundant.

### IPV-2 — Bulk accept implementation (per IP-5)
Reducer's `BULK_ACCEPT_PROPOSALS_FOR_*` is a no-op. Per IP-5, acceptance = `UPDATE_ASSET_FIELD` with the proposal value. So "Accept all suggestions" needs to:

- **(a)** **Dispatch the no-op `BULK_ACCEPT_PROPOSALS_FOR_GROUP` (telemetry hook) THEN loop `UPDATE_ASSET_FIELD` per asset per field in the same event handler.** React 19 batches synchronous dispatches in an event handler → 1 re-render. Recommended.
- (b) Loop only — skip the no-op dispatch. Saves one action but loses the telemetry-hook semantics IPI-1 intentionally preserved.
- (c) Enhance the reducer to actually do the bulk update — violates IP-5 + breaks the parity contract that says reducers don't auto-derive.
- **Default:** (a). Telemetry-friendly; performant under React 19 batching.

### IPV-3 — Regenerate behavior in C2.5
- **(a)** **UI-only stub.** Click ↻ → dispatch `REGENERATE_PROPOSAL` (reducer is no-op) → local component state shows "regenerating…" for 600ms then resets. Real regen lands at E2. Recommended.
- (b) Skip the regen UI entirely until E2. The icon appears later.
- (c) Synthesize a fake regen — replace the ghost text with a different mock value (random pick from a hardcoded pool). Demoware; risk of misleading.
- **Default:** (a). Confirms click landed; no fake content; matches the ACCEPT_PROPOSAL telemetry-hook pattern.

### IPV-4 — PriceBasisPanel mount points
Spec §9.3 says inline expansion below the price field. Spec §7.1 anatomy puts price-basis under "AI Proposal Detail" in the side panel.

- **(a)** **Both.** PriceBasisPanel is mounted in (1) AssetRow (Linear) inline below price field when `state.ui.priceBasisOpenAssetId === asset.id`, AND (2) inside the side panel's AI Proposal Detail "Price basis" expanded row. Same component, two mount points. Recommended.
- (b) Side panel only. Linear users have to open the panel to see basis.
- (c) Linear inline only. Side-panel users miss it.
- **Default:** (a)

### IPV-5 — Dev fixture seeding for AIProposalBanner
The banner is built and wired but `aiClusterProposals` is `[]` after hydration. To visually QA the banner in dev:

- **(a)** **Honor `?seedBanners=1` query param (dev-only).** When set, after hydration synthesize 1–2 `RECEIVE_AI_CLUSTER_PROPOSAL` dispatches using the first 1–2 storyGroups as proposal seeds. Rationale + confidence are mock strings. Mirrors `?simulateFailure=N` pattern from C2.4.
- (b) Always seed in dev mode (no opt-in needed).
- (c) Add a new scenario specifically for banner QA.
- **Default:** (a). Opt-in keeps the default dev experience matching production (banners only appear when AI pipeline pushes them).

### IPV-6 — AI Proposal Detail body — collapse mode
Three rows: Caption rationale / Price basis / Tag confidence.

- **(a)** **Independent collapse — multiple can be open simultaneously.** Each row has its own ▶/▼ toggle in local component state. Recommended for usability.
- (b) Accordion — only one open at a time. More visual discipline; less reading.
- (c) All open by default, all collapsible.
- **Default:** (a)

### IPV-7 — "Accept all suggestions" link in Linear (per-row)
Spec §10.2 mentions "per-row 'Accept all suggestions' link in Linear; bulk action in Compact+". Linear AssetRow currently has no such link.

- **(a)** **Yes — add a small "Accept all suggestions" link near the row header (top-right corner, next to ✕ Exclude).** Loops `UPDATE_ASSET_FIELD` for caption + tags + keywords (NEVER price) for this single asset. Recommended.
- (b) Skip — per-field ✓ icons already let user accept everything in 3-4 clicks.
- **Default:** (a). Explicit affordance reduces clicks, especially for clean fixtures where most suggestions are right.

### IPV-8 — Per-field ✓ disable when proposal is missing
What happens to a field whose `asset.proposal.<field>` is null/empty?

- **(a)** **Hide the ✓ icon entirely for that field.** No proposal → no affordance. Recommended (spec §9.1 implies icon only appears for fields with non-accepted suggestions).
- (b) Show a disabled ✓.
- **Default:** (a)

### IPV-9 — Test depth
Reducer state for AI proposals is comprehensively tested. New testable surfaces in C2.5: the bulk-accept loop pattern (per-event-handler dispatching N×UPDATE_ASSET_FIELD). This isn't really reducer-testable — it's a UI handler concern.

- **(a)** **No new reducer tests.** The action shapes are already covered. Visual smoke is the gate. Recommended.
- (b) Add a tiny reducer test that dispatches `BULK_ACCEPT_PROPOSALS_FOR_GROUP { fields: ['caption', 'tags'] }` followed by manual UPDATE_ASSET_FIELD asserts to document the C2.5 contract.
- **Default:** (a). The contract is documented in code comments + this directive; reducer behavior is a no-op which is already tested.

### IPV-10 — Title & geography ✓ in Linear
Spec §9.1 doesn't enumerate all fields explicitly. `asset.proposal` carries `title`, `description`, `tags`, `geography`. AssetRow currently has accept for description (caption) and price.

- **(a)** **Add ✓ for title, tags, geography too.** Pattern matches existing caption/price wiring. Recommended for completeness.
- (b) Tags only (skip title/geography — they're rarely ghost-suggested).
- **Default:** (a)

- AssetRow (Linear) renders ✓ accept icons next to title, caption, tags, geography, price fields when those fields have non-accepted proposal values

- StoryGroupAccordion "Accept all suggestions" button: dispatches `BULK_ACCEPT_PROPOSALS_FOR_GROUP` (no-op telemetry) THEN loops `UPDATE_ASSET_FIELD` for caption + tags + keywords across all assets in the cluster

- Dismiss banner dispatches `DISMISS_AI_CLUSTER_PROPOSAL` → banner hidden (already wired)
