## §1 What this is

The contextual action bar is the multi-select counterpart to the right-rail inspector. When the creator selects two or more assets in the contact sheet, the right-rail inspector unmounts (single-asset focus mode is meaningless on a multi-selection) and a floating action bar slides up from the bottom of the **center pane** (NOT the full window) exposing the operations that make sense in bulk.

Per the founding lock: bulk is **contextual on multi-select**. There is never a persistent toolbar. The bar exists only while `selectedAssetIds.length >= 2`.

## §5 Component anatomy

### §5.1 Visual layout (left → right)

```
┌─────────────────────────────────────────────────────────────────────────┐
│  12 SELECTED  │  Set price · Set privacy · Assign story · ✓ AI          │
│               │                              │  Compare · Exclude · Clear│
└─────────────────────────────────────────────────────────────────────────┘
```

Three logical groups separated by vertical-rule dividers:
1. **Count** — "{N} selected" left-aligned, bold uppercase, monospace number
2. **Field operations** — Set price · Set privacy · Assign story · ✓ AI
3. **Selection operations** — Compare · Exclude · Clear

Brutalist styling: 2px black border, white background, sharp corners, uppercase tracking-widest labels. Carry the `LABEL` / `BTN` style tokens from the dormant `BulkOpsBar.tsx` (verbatim copy into ContextualActionBar — DO NOT import).

### §5.2 Mount + position

- Parent: `CenterPane` (NOT shell-level)
- Position: `sticky bottom-0` inside the center pane's `flex flex-col` column, ordered above the footer row (ZoomSlider + CountFooter)
- Width: 100% of center pane (which is `flex-1` in the shell) minus small horizontal inset (`mx-4`) for visual breathing room
- z-index: `z-30` (above contact sheet content `z-10`, below any future modals)

### §5.3 Mount condition (inside ContextualActionBar)

```ts
if (state.ui.selectedAssetIds.length < 2) return null
if (state.commit.phase !== 'idle' && state.commit.phase !== 'summary') return null
```

Mirror inverse of the right-rail's `=== 1` gate. Both share the commit-phase guard — editing during commit is meaningless.

### §5.4 Slide-up animation

CSS transition on `transform: translateY()` and `opacity`. Use `useEffect` to set a `mounted` flag after first paint to trigger the transition from `translateY(100%) opacity-0` → `translateY(0) opacity-100` over `150ms`. Wrap the transition class application in `motion-safe:` so `prefers-reduced-motion: reduce` falls through to instant mount.

```jsx
className={`... transition-all duration-150 motion-reduce:transition-none ${
  mounted ? 'translate-y-0 opacity-100' : 'translate-y-full opacity-0'
}`}
```

## §6 Action specifications

### §6.1 Set price (popover)

- Anchor: button labeled `SET PRICE`
- Popover content:
  - Numeric input (EUR, accepts decimals to 2 places)
  - `APPLY` button (disabled when input empty or invalid)
- Apply behavior:
  - Convert EUR decimal → cents (multiply by 100, round)
  - Dispatch one `BULK_UPDATE_FIELD { assetIds, field: 'price', value: cents }`
  - Close popover
  - Selection persists (do NOT auto-clear)
- Carry the EUR-cents conversion math verbatim from `BulkOpsBar.tsx` lines 47-57

### §6.2 Set privacy (popover/dropdown)

- Anchor: button labeled `SET PRIVACY`
- Popover content: 3 buttons stacked vertically — `PRIVATE` / `RESTRICTED` / `PUBLIC`
- On click: dispatch `BULK_UPDATE_FIELD { assetIds, field: 'privacy', value: 'PRIVATE'|'RESTRICTED'|'PUBLIC' }` and close popover
- Selection persists

### §6.3 Assign story (popover)

- Anchor: button labeled `ASSIGN STORY`
- Popover content:
  - List of existing stories (`state.storyGroupOrder.map(id => state.storyGroupsById[id])`) — each clickable
  - Divider
  - Inline input + `+ NEW STORY` button — typing a name then clicking dispatches `CREATE_STORY_GROUP { name }` THEN moves the selection into the new story (must read the new story's id from the resulting state — see §6.3.1)
- Existing story click: for each selected assetId, dispatch `MOVE_ASSET_TO_CLUSTER { assetId, clusterId }`
- Selection persists (the assets are now grouped but still highlighted)

#### §6.3.1 New-story id resolution (IPD5-1)

**Decision required:** How does the bar learn the id of a freshly-created story so it can move assets into it?

Options:
- (a) Read state synchronously after `dispatch(CREATE_STORY_GROUP)` — works because `useReducer` returns the new state immediately, but our pattern dispatches imperatively from event handlers. The `state` in scope is stale until the next render.
- (b) Compute the id deterministically from `name` (slugify) — risky if names collide.
- (c) Two-step: `CREATE_STORY_GROUP` then trust the next render to expose it; do NOT auto-move in the same gesture. User must re-select the new story from the dropdown.
- (d) Add a `CREATE_STORY_GROUP_AND_MOVE { name, assetIds }` composite action to the reducer.

**Default = (d).** Composite action is the cleanest path; reducer authority preserved; no two-step UX. **Surface as Open Question §10.A — if rejected, fall back to (c).**

### §6.4 ✓ AI (button, no popover)

- Anchor: button labeled `✓ AI`
- Enabled when at least one selected asset has a non-null `proposal` field
- On click, executes the dispatch sequence returned by `computeAcceptAIDispatches(state, selectedAssetIds)`:
  1. One `BULK_ACCEPT_PROPOSALS_FOR_SELECTION { assetIds: <subset with proposals>, fields: ['caption', 'tags'] }` for telemetry no-op
  2. For each asset with a proposal, sequenced `UPDATE_ASSET_FIELD` writes for `caption`, `tags`, `geography` reading from `asset.proposal.{field}` — only fields where the proposal value is non-null AND the current asset value differs (don't overwrite manual edits with the same value)
- Selection persists

#### §6.4.1 Why a helper

The dispatch list is data; computing it is pure. Extracting it lets us unit-test the auto-accept logic without rendering React. Shape:

```ts
type AutoAcceptDispatch =
  | { type: 'BULK_ACCEPT_PROPOSALS_FOR_SELECTION'; assetIds: string[]; fields: ['caption', 'tags'] }
  | { type: 'UPDATE_ASSET_FIELD'; assetId: string; field: 'caption' | 'tags' | 'geography'; value: unknown }

function computeAcceptAIDispatches(
  state: V3State,
  selectedAssetIds: string[],
): AutoAcceptDispatch[]
```

#### §6.4.2 Test cases (5 minimum)

- 1 asset with full proposal (caption + tags + geography all non-null) → 1 BULK + 3 UPDATE
- 1 asset with no proposal → empty array (no dispatches)
- 2 assets, only one has proposal → 1 BULK (with that one assetId) + 3 UPDATE for that asset
- Mixed: 1 asset with proposal where caption matches current value → BULK + 2 UPDATE (skip caption)
- Asset has proposal but proposal.geography is null → BULK + 2 UPDATE (skip geography)

### §6.5 Compare (button)

- Anchor: button labeled `COMPARE`
- Enabled ONLY when `selectedAssetIds.length === 2` (per IPV4-3 default = a)
- Disabled state: dim + tooltip `Compare supports 2 assets at a time`
- On click: dispatch `ENTER_COMPARE_MODE { assetIds: state.ui.selectedAssetIds }`
- Note: D2.5 wires the dispatch only. The actual layout swap (Comparing layout state) is a D2.6 deliverable. Until D2.6 lands, the dispatch will set `compareAssetIds` in state but the center pane will still show the contact sheet — that's acceptable (no broken layout, just a state-only no-op visually).

### §6.6 Exclude (button)

- Anchor: button labeled `EXCLUDE`
- On click, for each selected asset NOT currently excluded, dispatch `TOGGLE_ASSET_EXCLUDED { assetId }` (so the action sets all selected assets to excluded; assets already excluded stay excluded — does NOT toggle them off — see IPD5-2)
- Selection persists (excluded assets are still selected; cards show via the `opacity-50` rule)

#### §6.6.1 IPD5-2 — Exclude behavior on mixed selection

**Decision required:** When the user has a mixed selection (some excluded, some not), what does Exclude do?

Options:
- (a) Toggle each independently (per spec literal) — produces inverted selection, surprising UX
- (b) Set all to excluded (default to "do the obvious thing")
- (c) Set all to included (inverse of b)
- (d) Show a sub-menu: "Exclude all" / "Include all"

**Default = (b).** Most predictable. The Exclude button always means "exclude". An "Include" / "Restore" affordance can be added later if needed. **Surface as Open Question §10.B.**

### §6.7 Clear (button)

- Anchor: button labeled `CLEAR`
- On click: dispatch `DESELECT_ALL_ASSETS`
- Bar slides down and unmounts on the next render

## §8 Right-rail interaction

The right-rail inspector mounts at `selectedAssetIds.length === 1`. The contextual action bar mounts at `selectedAssetIds.length >= 2`. These conditions are mutually exclusive; both cannot be on screen.

Transitions:
- 1 selected → 2 selected: inspector unmounts (right rail collapses), bar slides up
- 2 selected → 1 selected: bar slides down, inspector mounts
- 2 selected → 0 selected (Clear): bar slides down, no inspector
- 2 selected → 2 selected (selection set changes): bar count updates in place; no remount

The visual transition during selection-count changes is handled per-component (rail mount/unmount is instant; bar uses 150ms slide). No coordinated choreography needed.

## §9 Keyboard

- `Esc` with a popover open → close the popover, return focus to the popover's anchor button
- `Esc` with no popover open → dispatch `DESELECT_ALL_ASSETS` (bar dismisses)
- No new global shortcuts; selection shortcuts (Cmd-A, Shift-click, Cmd-click) are owned by the contact sheet (D2.3)

Per spec §14, popovers respond to `Esc`. Implementation: each popover registers a window keydown listener while open and unregisters on close.

## §10 Open questions (require ratification)

### §10.A — IPD5-1: How does Assign Story handle "+ New story"?

Options:
- (a) Read state synchronously after `dispatch(CREATE_STORY_GROUP)` — broken; state in handler closure is stale
- (b) Slugify name → id — collision risk
- (c) Two-step UX: create, then user re-selects the new story to move
- (d) NEW composite action `CREATE_STORY_GROUP_AND_MOVE { name, assetIds }`

**Recommended default: (d).** Cleanest UX, single-gesture creation+assignment, fits the reducer-authority lock. Adds one action type.

### §10.B — IPD5-2: What does Exclude do on mixed selection?

Options:
- (a) Toggle each (literal spec) — produces inverted selection, surprising
- (b) Set all to excluded
- (c) Set all to included
- (d) Sub-menu: Exclude / Include

**Recommended default: (b).** Predictable. Add "Include" / "Restore" later if asked.

### §10.C — IPD5-3: Slide-up animation timing

Options:
- (a) 150ms ease-out
- (b) 200ms ease-in-out
- (c) Instant (no animation)

**Recommended default: (a).** Fast enough to feel snappy, slow enough to register the appearance.

### §10.D — IPD5-4: Should the bar persist on filter chip change?

If the user has 5 assets selected and clicks a filter chip that hides 3 of them, does the bar:
- (a) Stay open with `length >= 2` of the still-visible selection
- (b) Stay open showing the full count (some selected assets are hidden)
- (c) Auto-clear selection on filter change

**Recommended default: (b).** Selection survives filter changes (per existing C2 behavior — `SET_FILTER_PRESET` doesn't touch `selectedAssetIds`). The bar shows total selected count regardless of visibility. Hidden selected assets are still operated on by bulk actions.

## §12 Risks

- **Animation jank on slow rerender** — if the bar slides up while CenterPane is mid-virtualization-scroll, the transition might stutter. Mitigation: use `transform` (GPU-accelerated) not `top`. Acceptable risk; punt if observed.
- **Popover positioning** — anchored popovers must clip within the center pane's horizontal bounds. If a popover anchored to the rightmost button (Clear) opens to the right, it'll overflow the center pane. Mitigation: align popovers to anchor's left edge by default; flip to right if not enough space. Use a `Portal`-free relative-positioned div for simplicity.
- **Concurrent state changes** — if the user clicks ✓ AI while the AI pipeline is still writing proposals server-side, the dispatched UPDATE_ASSET_FIELD could overwrite a not-yet-applied proposal. Acceptable: ✓ AI reads `asset.proposal` at the time of click; later proposal updates are independent.
