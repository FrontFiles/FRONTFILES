# D2.9 — Layout Polish + AI Provenance + Cover-in-Sheet

**Branch:** `feat/d-upload-v4`
**Spec refs:** UX-SPEC-V4 §3 / §4 / §5 / §11; D-PLAN (post-D2.8 polish phase, not previously enumerated).
**Status:** Awaiting ratification.

---

## §1 What this is

Three concurrent improvements to the upload workspace that together resolve the diagnosis "three neighboring systems, not one editorial workspace":

1. **Visual hierarchy reshape** — give each pane its own visual register (cognitive tempo) so the eye lands on the center, not on three competing surfaces.
2. **AI provenance tagging** — every editable field shows where its value came from (AI vs. creator) via a small inline tag. AI suggestions stay visible in the field by default, replaced by creator's value when they edit or approve.
3. **Cover slot in contact sheet** — when filtered to a story, the first visible cell IS the cover, blue-framed. Drop another asset onto it to set as cover. The inspector's `Set as cover` button is removed (drag is the canonical path).

D2.9 has both visual changes (lots of CSS) AND behavioral changes (reducer surface area for provenance). Founder lock L1 (reducer authority) preserved — no new action types, but `UPDATE_ASSET_FIELD` and `BULK_UPDATE_FIELD` reducers gain a side effect (set metadataSource), and `applyAIAutoAcceptSweep` does the same.

### Move 8 — Field provenance tagging (the AI visibility ask)

**State machine:**

Per editable field, the provenance is one of:
- `'ai'` — AI auto-accepted at hydration (caption / tags / geography) OR proposal value is being shown in the field because creator hasn't touched it yet (title / price / privacy / licences when they have a proposal)
- `'creator'` — creator typed in the field OR clicked the ✓ "approve AI" affordance
- `undefined` — no proposal exists AND creator hasn't typed (field is just empty)

**Display rules per field:**

```
if metadataSource[field] === 'creator':
  input value = editable[field]
  tag = "Edited by creator"      (slate-600, italic)

else if editable[field] is set (auto-accepted at hydration, source = 'ai'):
  input value = editable[field]
  tag = "AI generated"           (slate-500, italic)

else if proposal[field] exists:
  input value = proposal[field]  (rendered in slate-500 italic — visually muted)
  tag = "AI generated"           (slate-500, italic)

else:
  input empty
  no tag
```

**Reducer side effects (D2.9 changes):**

`UPDATE_ASSET_FIELD` reducer (v3-state.ts case):
```ts
case 'UPDATE_ASSET_FIELD': {
  const asset = state.assetsById[action.assetId]
  if (!asset) return state
  return {
    ...state,
    assetsById: {
      ...state.assetsById,
      [action.assetId]: {
        ...asset,
        editable: {
          ...asset.editable,
          [action.field]: action.value,
          metadataSource: {
            ...asset.editable.metadataSource,
            [action.field]: 'creator',  // ← NEW: any creator dispatch flips source
          },
        },
      },
    },
  }
}
```

`BULK_UPDATE_FIELD` reducer: same pattern — set `metadataSource[action.field] = 'creator'` for every asset in the loop.

`applyAIAutoAcceptSweep` (v3-hydration.ts):
```ts

  editable.description = asset.proposal.description
  editable.metadataSource = { ...editable.metadataSource, description: 'ai' }  // NEW
}
// Same for tags + geography.
```

**Approve flow:** the existing `[✓]` button (currently in InspectorAcceptRow) dispatches UPDATE_ASSET_FIELD with the proposal value. Our reducer change makes that dispatch ALSO flip metadataSource to 'creator'. So clicking ✓ does the right thing without any new action type.

**Reset flow (revert from creator → AI):** there is currently no spec for this. Defer. If creator wants to "go back to AI suggestion", they clear the field manually + click ✓ again, OR they regenerate (existing ↻).

**New component: `inspector/FieldProvenanceTag.tsx`** (~30 LOC):

```tsx

  source: 'ai' | 'creator' | undefined
  hasProposal: boolean
}

export default function FieldProvenanceTag({ source, hasProposal }: Props) {
  if (source === 'creator') {
    return <span className="text-[10px] italic text-slate-600">Edited by creator</span>
  }
  if (source === 'ai' || (source === undefined && hasProposal)) {
    return <span className="text-[10px] italic text-slate-500">AI generated</span>

  return null
}
```

### Move 9 — Cover slot in contact sheet (the cover-drag ask)

**Trigger:** ContactSheet's SORTABLE render path activates when `filter.storyGroupId !== null`. Move 9 only adds the cover slot in this path.

**Behavior:**

When the contact sheet is filtered to a story:
- The first cell of the grid IS the cover slot (a special card representing "this is the cover")
- If the story has a cover (`story.coverAssetId`), the slot displays that asset's thumbnail
- If no cover, the slot shows an empty box with hint text "Drop here to set cover"
- ALWAYS framed with `outline outline-4 outline-blue-600 outline-offset-0` + "COVER" pill top-left (`bg-blue-600 text-white text-[8px] px-1 py-0.5`)
- The slot is `useDroppable({ id: 'story-${storyId}-cover-slot' })`

**Drop routing:** `UploadShell.handleDragEnd` adds one new pattern after the existing `story-{id}-cover` match:

```ts
const coverSlotMatch = overId.match(/^story-(.+)-cover-slot$/)
if (coverSlotMatch) {
  const storyGroupId = coverSlotMatch[1]
  // Asset must be moved into the story IF it isn't already, then set as cover.
  for (const id of targetAssetIds) {
    const asset = state.assetsById[id]
    if (asset && asset.storyGroupId !== storyGroupId) {
      dispatch({ type: 'MOVE_ASSET_TO_CLUSTER', assetId: id, clusterId: storyGroupId })
    }
  }
  // Cover is single — only the dragged asset becomes cover (matches the
  // existing 'story-{id}-cover' left-rail pattern).
  dispatch({ type: 'SET_STORY_COVER', storyGroupId, assetId: activeId })
  return
}
```

**Visual treatment of the cover slot card** (new component `compare/CoverSlotCard.tsx` or inline in ContactSheet — simpler inline):
- Same dimensions as a regular contact-sheet card (16:9 at current cardSize)
- 4px blue outline always visible
- "COVER" pill top-left
- If cover exists: thumbnail of the cover asset, plus a subtle drop hint on `:hover` ("Drop to replace")
- If no cover: empty state with centered "Drop here to set cover"
- NOT clickable (selecting the cover slot does nothing — selection is a different concept)
- NOT draggable (cover slot itself isn't movable)

**ContactSheet rendering change:** in the SORTABLE path, prepend the cover slot to `visible[]` before passing to SortableContext. The slot has its own drop id and isn't part of the sortable items array (which is `visible.map(a => a.id)`). The cover slot is a non-sortable child rendered before the sortable grid.

Actually cleaner: keep the cover slot OUTSIDE the SortableContext, as a sibling at the top:

```tsx
{filter.storyGroupId && (
  <CoverSlot story={state.storyGroupsById[filter.storyGroupId]} />
)}
<SortableContext items={visible.map(a => a.id)} strategy={rectSortingStrategy}>
  <div className="grid ..." style={{ gap: 4 }}>
    {visible.map(asset => <SortableCellRenderer key={asset.id} asset={asset} ... />)}
  </div>
</SortableContext>
```

This keeps the sortable items array clean and doesn't try to make the cover slot sortable.

**SetAsCoverButton:** DELETED. The two paths to set cover post-D2.9:
- Drag onto left-rail story-header cover thumbnail (existing, kept)
- Drag onto contact-sheet cover slot when filtered to that story (NEW)

## §5 Open questions (require ratification)

### §5.A — IPD9-1: Where does FieldProvenanceTag render?

Options:
- (a) Below the field label, above the input
- (b) To the right of the field label, on the same line
- (c) Below the input field

**Recommended default = (a).** Sits in the natural reading flow: "this is the Title field; AI generated it; here's the value". (b) crowds the label. (c) creates two reading stops below each field.

### §5.B — IPD9-2: Cover slot when no cover is set yet

When `story.coverAssetId === null`, getStoryCover() falls back to the first asset in sequence. Should the cover slot:
- (a) Show that fallback cover (selector behavior — never empty visually)
- (b) Show an explicit empty state ("Drop here to set cover") even though a fallback exists

**Recommended default = (a).** Matches the current selector contract — the cover is "this story's cover" regardless of whether explicitly set. The drop affordance still works (drop another asset to make IT the explicit cover). Less confusing than empty-then-revealed-after-drop.

### §5.C — IPD9-3: Width math at narrow viewports

At 1280px viewport with left=320, right=400, center=560 → cards at zoom-3 (160 wide) = 3.5/row. Acceptable but tight. At 1024 viewport → center=304 → cards at 160 = 1.9/row (=1). Cramped.

Options:
- (a) Accept the tighter-on-narrow trade. Below 1024 viewport, creators collapse the left rail (existing leftRailCollapsed → 80px wide).
- (b) Make rail widths viewport-responsive (left: 320 ≥ 1280, 280 < 1280, 240 < 1024).
- (c) Cap rail widths but increase center pane padding so cards don't feel cramped.

**Recommended default = (a).** The collapse affordance already exists. Responsive rail widths add complexity for marginal benefit; this app targets desktop creators with ≥1280 displays.

### §5.D — IPD9-4: Approve `[✓]` button behavior on tags/geography (arrays)

Tags and Geography are arrays. AI suggests `['protest', 'lisbon', 'climate']`. Creator's editable might be `[]` (auto-accepted from AI, source = 'ai') or have manually-typed values (source = 'creator').

When source = 'ai' and creator clicks `[✓]` next to Tags: what happens?
- (a) Re-affirm. Dispatch UPDATE_ASSET_FIELD with the SAME tags array; source flips to 'creator'. Value unchanged but ownership transfers.
- (b) The `[✓]` is hidden for tags/geography because there's nothing to "approve" beyond what's already there.

**Recommended default = (a).** Consistent across all fields. Click ✓ = "I, creator, take ownership of this value." Same dispatch, same source flip.

## §7 Risks

- **Reducer change is the only behavioral risk.** Adding the metadataSource side effect to UPDATE_ASSET_FIELD could surface a parity test failure if any existing assertion checks the reducer output exhaustively. Mitigation: run v3-state-parity.test.ts; expect green (parity is between V2 and V3 SELECTORS, not reducer side effects). v3-state.test.ts may have brittle "exact state shape" assertions — those need updating to include `metadataSource`. Plan for it.
- **Width change to 320px** for left rail crowds the center on viewports <1280. Per IPD9-3 = (a), creators can collapse the rail.
- **Cover slot in SORTABLE mode** must not interfere with @dnd-kit's sortable strategy. The cover slot is a sibling of the SortableContext, NOT a sortable child. Verified pattern in §3 Move 9 details.
