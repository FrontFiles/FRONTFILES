### 3.0 Selector extraction (Option B amendment ‚Äî added 2026-04-26)

**Issue surfaced during pre-implementation audit:** the pure selectors that the parity contract depends on (`getAssetExceptions`, `getBlockingExceptions`, `getAdvisoryExceptions`, `getPublishReadiness`, `getTotalListedValue`, `getCompletionSummary`, plus the data-access helpers `getAssets`, `getIncludedAssets`, `getStoryGroups`, `getAnalysisProgress`) live INSIDE `v2-state.ts` and take `V2State` as input ‚Äî accessing fields like `state.batch.currentStage` and `state.ui.reviewEnteredEarly` that V3State (correctly) does not carry.

**Resolution (per founder ratification 2026-04-26):** Option B ‚Äî extract the selectors into a shared module before either reducer consumes them.

**New file: `src/lib/upload/upload-selectors.ts`**

Exports the surviving pure selectors with **narrow, explicit input types** ‚Äî never the full reducer state. Example shape:

```typescript

  storyGroupsById: Record<string, V2StoryGroup>

export interface DefaultsView {
  defaults: V2Defaults
}

export function getAssets(view: AssetsView): V2Asset[] { /* ... */ }
export function getIncludedAssets(view: AssetsView): V2Asset[] { /* ... */ }
export function getStoryGroups(view: StoryGroupsView): V2StoryGroup[] { /* ... */ }
export function getAssetExceptions(asset: V2Asset, defaults: V2Defaults): V2Exception[] { /* ... */ }
export function getBlockingExceptions(asset: V2Asset, defaults: V2Defaults): V2Exception[] { /* ... */ }
export function getAdvisoryExceptions(asset: V2Asset, defaults: V2Defaults): V2Exception[] { /* ... */ }
export function getPublishReadiness(view: AssetsView & DefaultsView): { ready: number; total: number } { /* ... */ }
export function getTotalListedValue(view: AssetsView): number { /* ... */ }
```

**Modifications to `v2-state.ts`** (not just dormancy header):
- Remove the surviving selector function bodies (the ones extracted above)
- Replace each with a thin re-export from `upload-selectors.ts` to preserve the existing import surface (so any dormant code that still imports them keeps working until cutover):
  ```typescript
  export { getAssets, getIncludedAssets, getStoryGroups, getAssetExceptions, /* ... */ } from './upload-selectors'
  ```
- `getExpressEligibility` does NOT extract (express path retired) ‚Äî stays in v2-state.ts as dormant
- The reducer function itself stays in v2-state.ts as dormant
- Dormancy header (per ¬ß3.3) goes ABOVE the imports, declaring the file dormant despite the re-exports

**Modifications to `v2-verification.ts`**:
- Update its imports to point at `upload-selectors.ts` instead of `v2-state.ts` for the moved functions
- This keeps DevHarness's verification flow working (per ¬ß11.5 + IP-7)

**No-behavior-change guarantee:** the extraction is mechanical. Function bodies move; imports redirect. Existing v2-state tests must continue passing without modification. This is the verification artifact for the extraction itself (no new tests; the existing v2-state tests are the regression net).

**Why narrow inputs (per founder guardrail):** accepting full `V2State` would re-import the stage-era assumptions into the shared module ‚Äî exactly the smell Option B was chosen to avoid. Narrow input types (`AssetsView`, `StoryGroupsView`, `DefaultsView`) make the selector contract explicit about what it actually reads. Both V2State and V3State satisfy them via structural typing without either side knowing about the other's full shape.

#### `src/lib/upload/v3-state.ts` (new)

Exports:
- `V3Action` (re-export from v3-types for ergonomics)
- `v3InitialState(batchId: string): V3State`
- `v3Reducer(state: V3State, action: V3Action): V3State` ‚Äî pure function, exhaustive switch over V3Action

Reducer requirements:
- Exhaustive switch with TypeScript `never` check at the default case
- Pure (no side effects, no randomness, no current-time except where explicitly part of action payload)
- For every action in `C2-PLAN.md` ¬ß5.2 (KEEP) and ¬ß5.3 (NEW), produce the documented state transition
- Reject malformed actions with explicit guards (e.g., `BULK_ACCEPT_PROPOSALS_FOR_GROUP` with `fields` array containing `'price'` MUST be rejected ‚Äî throw `Error('bulk_accept_price_forbidden')` per spec ¬ß9.2)

Key transitions (non-exhaustive ‚Äî full list derived from ¬ß5):
- `OPEN_SIDE_PANEL`: sets `ui.sidePanelOpenAssetId`; preserves selection
- `CLOSE_SIDE_PANEL`: clears `ui.sidePanelOpenAssetId`; preserves selection
- `NAVIGATE_SIDE_PANEL`: moves `ui.sidePanelOpenAssetId` to next/prev within filtered visible assets per IP-8
- `BEGIN_COMMIT`: `commit.phase: 'idle' ‚Üí 'summary'`; allows further field edits per IP-9 (no UI lock)
- `CANCEL_COMMIT`: `commit.phase: 'summary' ‚Üí 'idle'`
- `CONFIRM_COMMIT`: `commit.phase: 'summary' ‚Üí 'committing'`; initializes `perAssetProgress = {}`
- `COMMIT_PROGRESS_UPDATE`: updates `commit.perAssetProgress[assetId]`
- `COMMIT_SUCCEEDED`: `commit.phase: 'committing' ‚Üí 'success'`
- `COMMIT_PARTIALLY_FAILED`: `commit.phase: 'committing' ‚Üí 'partial-failure'`; sets `commit.failed`
- `RETRY_FAILED_COMMITS`: `commit.phase: 'partial-failure' ‚Üí 'committing'`; clears `commit.failed`; reinitializes per-asset progress for failed-only
- `RESET_FLOW`: returns to `v3InitialState(state.batch.id)`
- `ACCEPT_PROPOSAL`: NO STATE CHANGE per IP-5 (selector-derived); kept as a no-op action for telemetry hooks (or omitted ‚Äî see ¬ß6 IP)
- `REGENERATE_PROPOSAL`: dispatches a re-fetch (mock simulation engine integration in C2.5); for C2.1, a no-op
- `RECEIVE_AI_CLUSTER_PROPOSAL`: appends to `aiClusterProposals` with `status: 'pending'`
- `ACCEPT_AI_CLUSTER_PROPOSAL`: marks proposal `status: 'accepted'`; creates Story group from proposed asset IDs
- `DISMISS_AI_CLUSTER_PROPOSAL`: marks proposal `status: 'dismissed'`
