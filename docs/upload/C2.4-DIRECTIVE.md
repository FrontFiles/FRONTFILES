## 4. Implementation Points (ratify before implementation)

### IPIV-1 â€” Fake-progress driver location
The reducer accepts `COMMIT_PROGRESS_UPDATE { assetId, progress }` and `COMMIT_SUCCEEDED`. Something has to dispatch them after `CONFIRM_COMMIT` to drive the `committing â†’ success` transition. Real network = PR 5; for C2.4 this is fake.

- **(a)** Custom hook `useCommitSimulation()` mounted in `CommitBar`. Watches `state.commit.phase`; when it transitions to `committing`, schedules a series of `setTimeout` calls dispatching `COMMIT_PROGRESS_UPDATE` for each included asset and a final `COMMIT_SUCCEEDED`. Cleans up on unmount or phase change. Recommended â€” testable in isolation, separates concern from the bar.
- (b) Inline `useEffect` inside `CommitBar`. Smaller surface but harder to extract for tests later.
- (c) Reuse `src/lib/upload/v2-simulation-engine.ts`. It exists and was designed for V2; coupling C2.4 to it pulls in V2-coupled lifecycle assumptions and risks regression.
- **Default:** (a)

### IPIV-2 â€” Success-phase top-level branch in UploadShell
Spec Â§11.3 L6 says success replaces the entire screen body.

- **(a)** Add a top-level branch in `UploadShell.tsx`: `if (state.commit.phase === 'success') return <CommitSuccessPanel />`. Skip rendering drop zone + session defaults + asset-list/panel row + commit bar. Recommended.
- (b) Render the regions but visually hide them with CSS. Less clean.
- **Default:** (a)

### IPIV-3 â€” Error-phase rendering scope
Per L7, partial-failure keeps the asset list visible; only the bottom bar swaps.

- **(a)** `CommitBar` orchestrator renders `<CommitErrorPanel />` when `phase === 'partial-failure'`. Asset list, drop zone, session defaults stay live (so user can edit a failed asset and retry). Recommended.
- (b) Replace whole screen like success. Spec rejects this.
- **Default:** (a)

### IPIV-4 â€” Simulation timing
How fast does the fake `committing â†’ success` run?

- **(a)** Per-asset tick of 100ms; one progress event per asset bringing it from 0â†’100 in 1 step (simplest). Total time = `includedCount Ă— 100ms`, capped at 5s by spreading ticks if N > 50. Recommended for visual demo speed.
- (b) Animate per-asset 0â†’100 over 500ms with intermediate ticks. More realistic; more code.
- (c) Single bulk wait then SUCCESS. Boring; doesn't exercise progress UI.
- **Default:** (a). Easy to swap to (b) later.

### IPIV-5 â€” Dev-only failure injection
For visual smoke of the partial-failure panel, we need a way to trigger `COMMIT_PARTIALLY_FAILED` without a real network failure.

- **(a)** Honor a `?simulateFailure=N` query param (dev-only, gated by `process.env.NODE_ENV === 'development'`). The simulation hook reads it on mount; if set, dispatches `COMMIT_PARTIALLY_FAILED` with the first N included assets after the progress finishes. Recommended.
- (b) Keyboard shortcut (e.g., `Cmd+Shift+F`) inside CommitProgressPanel.
- (c) Skip â€” only ever exercised at PR 5 with real network.
- **Default:** (a). Lets you visually QA the error panel today via `?scenario=...&simulateFailure=2`.

### IPIV-6 â€” Plain-language summary text helper
Spec Â§10.2 table maps `(readyCount, blockerCounts)` â†’ user-facing string.

- (b) Inline the logic in `CommitBar`. Smaller blast radius but harder to test.
- **Default:** (a)
