## 3. Polish items

### P1 — SideDetailPanel thumbnail height cap
**Where:** `SideDetailPanel.tsx` `<Thumbnail />`
**Today:** `aspect-square` × `w-full` on a 480px panel = 480px tall thumbnail. Eats the visible viewport; field editor scrolls below the fold.
**Change:** cap height. Default: `max-h-48` (192px) with `aspect-square` removed; image uses `object-contain` inside a fixed-height frame. Filename + thumbnail readable above the fold; fields section visible without scrolling.
**Severity:** Must-fix (blocks C2.3 visual gate).

### P2 — BulkOpsBar visual hierarchy when narrow
**Where:** `BulkOpsBar.tsx`
**Today:** Already uses `flex-wrap`. When asset list shrinks (panel open), the bar wraps but all elements equal-weight, so it reads as visual noise.
**Change:** Add a thin vertical separator between groups (selected count · price group · privacy group · apply-to group · clear). Reduce gap when wrapped (`gap-x-3 gap-y-2`). When wrapped to 2+ rows, group "Apply to selected:" into a sub-cluster.
**Severity:** Should-fix (visible on every Batch/Archive scenario when panel opens).

### P3 — "0 READY" cluster header math: ACCEPT current behavior
**Where:** `StoryGroupAccordion.tsx` line 40-46
**Today:** Already correctly filters `needs_story` per Phase 7e. The "0 READY" on `archive_150_mixed` is **truthful** — assets at hydration are missing privacy / price / licences, hence blocked. After clicking "Accept all suggestions" + setting privacy + price, the count rises. No bug.
**Change:** **No code change.** Add a one-line comment in the cluster header explaining why "0 READY" is accurate, so future debugging doesn't go hunting.
**Severity:** Documentation-only.

### P4 — SideDetailPanel sticky header z-index audit
**Where:** `SideDetailPanel.tsx` `<PanelHeader />`
**Today:** Header has `sticky top-0 bg-white z-10` — verify it doesn't conflict with the sticky CommitBar at the bottom (`z-20`). Also verify side-panel content scrolling doesn't push the panel itself.
**Change:** Verify only. If conflict found, raise CommitBar to `z-30` or lower header to `z-5`. Likely no change needed.
**Severity:** Verification-only.

### P5 — Per-row "Accept all suggestions" link weight
**Where:** `AssetRow.tsx` row header
**Today:** `text-blue-600` text link, no border. Reads as secondary against the brutalist button styling everywhere else.
**Change:** Add a border to match the brutalist convention: `border border-blue-600 px-2 py-0.5 hover:bg-blue-600 hover:text-white transition-colors` while keeping text-blue-600 base. Matches the cluster-header bulk action visual weight without competing with the per-field ✓ buttons (which are black-bordered).
**Severity:** Should-fix (consistency).

### P6 — AcceptRow vertical vs horizontal in side panel
**Where:** `SideDetailPanel.tsx` `AcceptRow` component
**Today:** ✓ + ↻ stack vertically (`flex-col gap-1`). Reasonable on a 480px panel where horizontal space is tight, but unnecessarily tall — pushes fields apart and adds visual rhythm noise.
**Change:** Switch to horizontal (`flex-row gap-1`). The field-editor flex container is already `items-start gap-1.5` so the buttons sit beside the input top-aligned.
**Severity:** Should-fix.

### P7 — Drop zone + session defaults placeholder text styling
**Where:** `UploadShell.tsx` lines 104 + 117 (Region 1 + the session defaults bar)
**Today:** Renders `Drop zone — placeholder for C2.2 (currently archive mode, 150 files)` and `Session defaults bar — placeholder for C2.1 (collapsed: no)` as small slate-400 text.
**Change:** Depends on IPVI-1. If placeholders stay (C2.7 deferral), the text is fine — verify it's not visually noisy at the brutalist level.
**Severity:** Tied to IPVI-1.

## 4. Audit dimensions (the brutalist principles being applied)

These apply across every change in §3 and any judgment call mid-implementation:

| # | Principle | Spot-check |
|---|---|---|
| 1 | Black borders, square corners, no shadows | All surfaces inspected — pass. Verify CommitSummaryPanel + CommitErrorPanel don't introduce drift |
| 2 | Black / blue-600 / white only (slate accents permitted for muted text and chip backgrounds) | Already consistent across 9 components per grep |
| 3 | Strong hierarchy: primary action visually dominant | CommitBar CTA = bg-blue-600, RETRY FAILED in error panel = bg-blue-600. Per-row Accept-all link is the gap (P5 fix) |
| 4 | No decorative noise — every visual element earns its presence | Per-row Accept-all link reads as decoration today (P5); thumbnail eating panel viewport reads as filler (P1) |
| 5 | Tight uppercase 10px tracking-widest for labels; mono for numerics | Consistent across components |
| 6 | Restraint with color — yellow only for warnings, red only for failures | yellow-300 (chips) + yellow-50 (CommitErrorPanel + ConflictResolver) + red-600 (commit-failed chip) — pass |
| 7 | No emojis except the explicit ✓ ↻ ✕ ▶ ▼ ⚠ ℹ — verified | Pass |

If a change in §3 introduces drift on any audit dimension, halt and surface.

## 5. Implementation Points (ratify before implementation)

### IPVI-1 — DropZone + SessionDefaultsBar — implement, defer, or skip?
The two real placeholders. C2-PLAN.md §4.1 lists them as separate components. They were never built because C2.1 was reducer-focused and C2.2 was asset-list-focused. PR #14 ships the entire C2 work but these surfaces remain text placeholders.

- **(a)** **Implement minimal stubs in C2.6.** DropZone gets a real `<input type="file" multiple>` + drag-drop listener that dispatches `ADD_FILES`. SessionDefaultsBar gets a collapsible region with privacy / licences / tags / watermark inputs that dispatch `SET_DEFAULTS`. ~150 lines for both. Lands inside C2.6 commit.
- (b) **Defer to C2.7.** Compose a separate directive after C2.6 polish ships. Cleaner separation; PR #14 stays "structural rebuild" without functional drop-zone.
- (c) **Skip — placeholders stay until PR 5 backend wiring.** Real upload flow needs the file-handling backend anyway. Document the placeholders as "shell only — wired at PR 5".
- **Default: (b) — defer to C2.7.** Reasoning: C2.6 is for *polish*, not implementation. Mixing placeholder implementation into a polish commit muddies the diff. C2.7 can be its own focused 1-commit phase. (c) is also defensible if the founder wants PR 5 to own the file flow end-to-end.
