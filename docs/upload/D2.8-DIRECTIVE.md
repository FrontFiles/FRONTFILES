## §1 What this is

The final cleanup PR for the V4 rebuild. After D2.1–D2.7 + D2.5 + D2.6 landed, the upload module has two parallel folders living side-by-side:

```
src/app/vault/upload/
  _components/        ← C2 spine — mix of dormant (8 files) and surviving (10 files)
  _components_v4/     ← V4 implementation — 32 files, sole production code path
```

D2.8 collapses this back to ONE folder by:

1. Deleting the 8 dormant C2 components (already flagged for D2.8 deletion in their headers)
2. Moving all 32 V4 files into the now-thinned `_components/` folder
3. Updating import paths
4. (Optional, recommended) Renaming `UploadShellV4` symbol → `UploadShell`

After D2.8, the upload tree is a single clean module with no V4 suffix anywhere.

## §7 Open questions (require ratification)

### §7.A — IPD8-1: Rename `UploadShellV4` → `UploadShell`?

Options:
- (a) Rename. The V4 suffix is meaningless once V4 is the only shell. Symbol cleanliness, no extra cognitive load for future readers.
- (b) Keep `UploadShellV4`. The git history is cleaner (no symbol rename), and renaming is purely cosmetic.

**Recommended default = (a) rename.** This IS the cutover — leaving the V4 suffix would be a half-finished cleanup. Worth the small amount of import-update churn.

### §7.B — IPD8-2: Commit-bar cluster (6 files) — keep or delete?

The CommitBar + 4 panels + useCommitSimulation cluster is currently orphaned in production code: only the dormant `UploadShell.tsx` (about to be deleted) imports it. UploadShellV4 has a placeholder slot:

```tsx
{/* Commit bar slot — placeholder for the D-phase that re-mounts CommitBar.
 *
 * The C2.4 commit bar (CommitBar.tsx + 4 sub-panels + useCommitSimulation
 * hook) carries forward unchanged in shape and behavior per D-PLAN §4.2.
 * A subsequent D phase remounts it here as a sibling of the three-pane row. */}
```

Options:
- (a) **Keep all 6 files** as future-use spine carryovers. The slot exists, the re-mount is planned, no behavioral regression. After D2.8 the cluster is technically dead code until a later D-phase re-mounts it.
- (b) **Also delete the cluster** as part of D2.8. The re-mount D-phase would re-write or re-import them later. Aggressive but defensible — dead code is dead code.
- (c) **Stage the re-mount in D2.8 itself** — move the cluster + wire it up in UploadShell.tsx. Bigger PR, mixes concerns.

**Recommended default = (a) keep.** The cluster is unchanged-shape spine code per D-PLAN §4.2. The placeholder slot in UploadShell is the contract. Deleting and re-creating later loses git blame and risks behavioral drift. (a) preserves the future-use intent at zero risk. The cluster shows up in the post-D2.8 tree (§5) as "spine carryover, slot-mounted (pending re-wire)".

### §7.C — IPD8-3: Should D2.8 also clean up references to "V4" in surviving comments?

There are comments throughout the surviving spine files (and some V4 files) that reference V4 as a parallel track. Examples: `D2.7 IPD7-13 = (a)`, `Per UX-SPEC-V4 §3.4`, etc. These are mostly fine because UX-SPEC-V4 is still the canonical spec name.

Options:
- (a) Leave comments alone. They reference the spec/directive names which are authoritative artifacts.
- (b) Sweep comments to drop "V4" qualifiers where they're internal-only (e.g., `D2.7 IPD7-13 = (a)` is fine because IPDs are versioned).
- (c) Aggressive comment normalization.

**Recommended default = (a) leave alone.** Comments reference real artifacts (UX-SPEC-V4, D-PLAN, IPDs). Renaming them would obscure the historical trail. Comment polish is not D2.8's concern.

### §7.D — IPD8-4: Single PR or split?

The cutover involves: 8 deletions + 32 moves + ~40 import edits + 1 symbol rename. That's a chunky diff.

Options:
- (a) **Single commit** — clean, atomic, matches the "cutover" framing. Diff is large but mechanical.
- (b) **Two commits**: (1) `git rm` dormant C2; (2) `git mv` V4 → C2 + import updates. Slightly easier review.
- (c) **Three commits**: (1) delete dormant; (2) move folder; (3) update imports.

**Recommended default = (a) single commit.** D2.8 is "the cutover" — semantically one operation. git mv preserves blame on each file. Reviewers can use `--find-renames` to follow the moves. The mechanical bulk is a feature, not a bug.

## §9 Risks

- **Mass-import churn risk** — 32 moved files × ~3-5 imports each = ~100 import-line edits. A single typo in a relative path breaks compilation. Mitigation: TypeScript catches these immediately on first compile; the script verifies with `bunx tsc --noEmit` before commit.
- **git mv preservation** — must use `git mv` (not delete + add) so blame survives. Mitigation: script uses `git mv` explicitly, then verifies blame is intact on a sample file.
- **Hot-reload weirdness** — Next.js / Turbopack caches may serve stale modules during/after the move. Mitigation: kill dev server, `rm -rf .next`, restart. Script doesn't auto-restart the dev server; founder restarts manually.
- **Dormant C2 file un-deletion** — git history preserves the deleted files; rollback is `git revert <D2.8-commit>`. No data loss.

## §10 Rollback

Single-commit revert:

```
git revert <D2.8-sha>
```

Restores both the deleted C2 files AND the `_components_v4/` folder structure (since git revert undoes mv operations correctly).

If the diff is too large to revert cleanly (unlikely — mechanical mv shouldn't conflict), the per-file restore path is:

```
git checkout <pre-D2.8-sha> -- src/app/vault/upload/_components/   src/app/vault/upload/_components_v4/   src/app/vault/upload/page.tsx
```
