## 1. What F1.5 is

F1.5 is the small bridge between F1 (architecture) and F2 (schema migration). It produces three artifacts:

1. **Five empty CSV templates** (v3 row counts — was 63/63/17/12/7 in v2; was two CSVs in v1) — `format_defaults_v1_eur.csv` (**21 rows full scope; 7 rows v1 editorial-only Stage B scope**) + `platform_floors_v1_eur.csv` (21 rows full scope; 7 rows v1 ed-only) + `sublabel_multipliers_v1.csv` (17 rows; 6 v1 ed-only) + `use_tier_multipliers_v1.csv` (12 rows; 4 v1 ed-only) + `cc_variants_v1_eur.csv` (7 rows; flat per variant) — all generated by the bootstrap script
2. **The calibration process** — what sources to consult, how to fill, what to validate, which cells need extra care
3. **A converter script** — reads the founder-filled CSVs and emits a SQL seed migration (5 INSERT blocks; one per table) that F2 includes verbatim

After founder fills the CSVs and the converter produces the SQL seed:
- F2 schema migration directive composes against a fully-resolved table content
- Founder reviews the SQL seed migration before F2 merges
- F3 (`format_defaults` adapter implementation) ships against ratified data

F1.5 ships dormant — no engine code; no migration; no UI. It's a tooling + spec PR. The actual calibration is founder offline work.

|---|---|---|

### 6.3 Per-cell guidance (founder offline reference)

The 21 cells (full scope) aren't independent — there are intuitive ratios across dimensions. Founder may calibrate by anchoring a few "spine" cells and deriving others by ratio. The directive does NOT prescribe ratios — that's editorial judgment — but offers the spine-cell anchoring approach as a possible workflow:

| Spine cell candidate (v3) | Why anchor here |
|---|---|
| `photo / editorial` | The most common asset shape on Frontfiles; market data is densest here; anchor first. v1 ed-only Stage B drop-in: €220 HIGH per CANONICAL-SOURCES.md §11.2 |
| `photo / commercial` | Defines the editorial → commercial spread (deferred to v2 per Stage B scope) |
| `photo / advertising` | Defines the commercial → advertising spread — advertising commands premium (deferred to v2 per Stage B scope) |

Once 1-3 spine cells are anchored, the remaining cells can be derived by ratio + sanity-check. F1 §3.3 calibration sources guide the spine values; **CANONICAL-SOURCES.md** Phase α 5-tier source authority hierarchy + per-format source registry covers the v1 editorial scope already. **CC pricing is calibrated separately in `cc_variants_v1_eur.csv` per §6A** (absolute prices per variant).

### 6.4 Validation rules (enforced by the converter; v3)

- All 21 rows present (no missing combinations)
- `format`, `licence_class` values match the enumerated set (no typos, no bonus values)
- `licence_class` ∈ `{editorial, commercial, advertising}` only — `creative_commons` REJECTED here (belongs in `cc_variants_v1_eur.csv`)
- **v3 corrigendum:** any leftover `intrusion_level` column REJECTED — the v3 schema has 5 columns (`format,licence_class,currency,baseline_cents,calibration_basis`); a 6-column CSV indicates the founder is filling a stale v2-shape template and must regenerate via bootstrap
- `currency` = `EUR` on every row
- `baseline_cents` is a positive integer (no nulls, no zeros, no decimals)
- `baseline_cents` is within sanity bounds: 100 ≤ value ≤ 10000000 (€1.00 to €100,000.00); flagged if outside, hard-error if < 0
- `calibration_basis` is non-empty (≥ 5 chars); helps recalibration teams understand the row's source
- No duplicate `(format, licence_class)` combinations

The converter prints validation warnings + errors before generating SQL. Errors block; warnings surface for founder review.

### 6A.3 Per-cell guidance

CC0 = 0 is canonical. The other 6 variants encode increasing rights restriction:
- CC-BY: most permissive (attribution only); typical small admin fee
- CC-BY-SA: attribution + share-alike; similar to CC-BY in price
- CC-BY-ND: attribution + no derivatives; modest premium (no remixing)
- CC-BY-NC: attribution + non-commercial; commercial-use buyers pay; non-commercial use is free under CC
- CC-BY-NC-SA / CC-BY-NC-ND: most restrictive; commercial-use buyers pay highest CC tier

```csv
illustration,editorial,EUR,
... (17 more rows — bootstrap script generates all 21)
```

### 7.3 Per-cell guidance (v3)

Floors should sit **strictly below** the corresponding `format_defaults` cell at the lowest use_tier multiplier. Specifically: for any `(format, licence_class)` cell in `platform_floors`, the `min_cents` should be ≤ `baseline_cents` × `min(use_tier_multiplier across 4 tiers)` for the matching `(format, licence_class)` cell in `format_defaults`. The converter's cross-CSV check (§7.4) enforces this as a warning.

If floors are set higher than format_defaults, the engine clamps every recommendation up — meaning format_defaults effectively never applies. That's almost always wrong.

The converter flags floor > min(defaults) as a warning (not an error) so founder can decide.

### 7.4 Validation rules

Same shape as §6.4:
- All 21 rows present
- Enumerated values match (`format` ∈ 7 enum + `licence_class` ∈ 3 enum)
- **v3 corrigendum:** any leftover `intrusion_level` column REJECTED — the v3 schema has 4 columns (`format,licence_class,currency,min_cents`); a 5-column CSV indicates a stale v2-shape template
- `currency` = `EUR`
- `min_cents` positive integer; sanity-bounded 100 ≤ x ≤ 10000000
- No duplicate `(format, licence_class)` combinations
- Cross-CSV check: `min_cents` ≤ `min(baseline_cents across use_tiers)` for matching `(format, licence_class)` — warning, not error

### 9.3 Why a bootstrap script (not just commit empty CSVs directly)

The dimension enumerations might drift between F1.5 composition and F2 ship. Bootstrapping from the actual `AssetFormat` enum + the documented dimension lists keeps the CSVs aligned with code. If `AssetFormat` adds a new value (e.g., `3d`), re-running bootstrap regenerates the CSV with the additional rows for the new format — founder fills only the new cells.

The bootstrap reads the canonical enumeration from `src/lib/types.ts` if available; falls back to the literal lists in §9.1 if the file doesn't have an exported enum yet. Either way the source-of-truth-divergence risk is minimized.

## 10. Converter script — `scripts/pricing/csv-to-seed-migration.ts`

### 10.1 Behavior (v3)

Reads the **five** filled CSVs and emits a SQL seed migration with **5 INSERT blocks**. Validation runs first; errors block; warnings print but allow continuation. v3 vs v2: the `format_defaults` and `platform_floors` INSERT shapes drop the `intrusion_level` column from both the column list AND the per-row builder (per L1 v2.2 §5.0; F1 v4 §3.1 + §4.1 schemas).

```typescript
// Pseudocode (v3)

async function main() {
  const defaultsCsv = readCsv('docs/pricing/calibration/format_defaults_v1_eur.csv')           // 21 rows (v3; was 63)
  const floorsCsv = readCsv('docs/pricing/calibration/platform_floors_v1_eur.csv')             // 21 rows (v3; was 63)
  const sublabelMultCsv = readCsv('docs/pricing/calibration/sublabel_multipliers_v1.csv')      // 17 rows
  const useTierMultCsv = readCsv('docs/pricing/calibration/use_tier_multipliers_v1.csv')       // 12 rows
  const ccVariantsCsv = readCsv('docs/pricing/calibration/cc_variants_v1_eur.csv')             // 7 rows

  // Validation pass — per-CSV
  const errors = []
  const warnings = []
  validateRows(defaultsCsv, /* §6.4 */, errors, warnings)
  validateRows(floorsCsv, /* §7.4 */, errors, warnings)
  validateRows(sublabelMultCsv, /* §6B.4 */, errors, warnings)
  validateRows(useTierMultCsv, /* §6C.4 */, errors, warnings)
  validateRows(ccVariantsCsv, /* §6A.4 */, errors, warnings)
  validateFloorsBelowDefaults(floorsCsv, defaultsCsv, useTierMultCsv, warnings)  // §7.3 cross-check (uses min use_tier multiplier; matched on (format, licence_class) per v3)
  validateCc0IsZero(ccVariantsCsv, warnings)  // §6A.4 — CC0 should be 0
  if (errors.length > 0) {
    console.error('VALIDATION ERRORS — converter aborted:')
    errors.forEach(e => console.error('  ' + e))
    process.exit(1)
  }
  if (warnings.length > 0) {
    console.warn('Warnings (proceeding anyway):')
    warnings.forEach(w => console.warn('  ' + w))
  }

  const tableVersion = 1
  const calibratedAt = new Date().toISOString()
  const calibratedBy = process.env.CALIBRATED_BY_USER_ID

  if (!calibratedBy) {
    console.error('CALIBRATED_BY_USER_ID env var required')
    process.exit(1)
  }

  let sql = `-- F2 seed (v3): 5 pricing tables (EUR); cascade C per L1 v2.2 §5.0\n`
  sql += `-- Generated by scripts/pricing/csv-to-seed-migration.ts on ${calibratedAt}\n\n`

  // Block 1 — pricing_format_defaults (21 rows; v3: no intrusion_level column; no use_tier column)
  sql += `INSERT INTO pricing_format_defaults\n`
  sql += `  (format, licence_class, currency, baseline_cents, table_version, effective_from, calibration_basis, calibrated_at, calibrated_by)\n`
  sql += `VALUES\n`
  sql += defaultsCsv.map(r =>
    `  ('${r.format}', '${r.licence_class}', '${r.currency}', ${r.baseline_cents}, ${tableVersion}, now(), ${quoteString(r.calibration_basis)}, now(), '${calibratedBy}')`
  ).join(',\n') + ';\n\n'

  // Block 2 — pricing_platform_floors (21 rows; v3: no intrusion_level column)
  sql += `INSERT INTO pricing_platform_floors\n`
  sql += `  (format, licence_class, currency, min_cents, effective_from, calibrated_at, calibrated_by)\n`
  sql += `VALUES\n`
  sql += floorsCsv.map(r =>
    `  ('${r.format}', '${r.licence_class}', '${r.currency}', ${r.min_cents}, now(), now(), '${calibratedBy}')`
  ).join(',\n') + ';\n\n'

  // Block 3 — pricing_sublabel_multipliers (17 rows; per L2 §6.3)
  sql += `INSERT INTO pricing_sublabel_multipliers\n`
  sql += `  (sublabel, multiplier, effective_from, calibrated_at, calibrated_by)\n`
  sql += `VALUES\n`
  sql += sublabelMultCsv.map(r =>
    `  ('${r.sublabel}', ${r.multiplier}, now(), now(), '${calibratedBy}')`
  ).join(',\n') + ';\n\n'

  // Block 4 — pricing_use_tier_multipliers (12 rows; per L2 §6.4)
  sql += `INSERT INTO pricing_use_tier_multipliers\n`
  sql += `  (licence_class, use_tier, multiplier, effective_from, calibrated_at, calibrated_by)\n`
  sql += `VALUES\n`
  sql += useTierMultCsv.map(r =>
    `  ('${r.licence_class}', '${r.use_tier}', ${r.multiplier}, now(), now(), '${calibratedBy}')`
  ).join(',\n') + ';\n\n'

  // Block 5 — pricing_cc_variants (7 rows; per L2 §6.2)
  sql += `INSERT INTO pricing_cc_variants\n`
  sql += `  (cc_variant, currency, price_cents, effective_from, calibrated_at, calibrated_by)\n`
  sql += `VALUES\n`
  sql += ccVariantsCsv.map(r =>
    `  ('${r.cc_variant}', '${r.currency}', ${r.price_cents}, now(), now(), '${calibratedBy}')`
  ).join(',\n') + ';\n'

  writeFile('supabase/migrations/_pending/<TIMESTAMP>_pricing_seed_v1.sql', sql)
  console.log('Wrote SQL seed (5 blocks; 78 total rows full scope, or 31 v1 editorial-only) to _pending/.')
}
```

### 10.2 Output location

Generated SQL lands in `supabase/migrations/_pending/` (a directory the converter creates if absent). F2 directive will move it to `supabase/migrations/` with a timestamp prefix at F2 ship time.

The `_pending/` directory is gitignored or kept out of the migration runner's path so accidental untimed-migration application doesn't happen.

### 10.3 Re-run semantics

Re-running the converter with edited CSVs regenerates `_pending/` SQL with `table_version = 1`. **The version bump is manual** — when calibration is recalibrated post-launch, the founder bumps the version constant in the converter script (or passes `--version=2` flag) and the new SQL preserves old rows + adds new rows per F1 §3.3 recalibration cadence.

For v1 seed: `table_version = 1` is the only value the converter produces.

### 10.4 SQL escaping

The `calibration_basis` strings may contain quotes, commas, etc. The converter uses parameterized SQL OR proper SQL escaping (`E'...'` strings with backslash escapes; or `$$...$$` dollar-quoting). The reference implementation must handle:
- Single quotes in basis text (e.g., "fotoQuote's 2024 editorial midpoint")
- Newlines in basis text (rare; escape as `\n`)
- Unicode in basis text (UTF-8 input → UTF-8 output)

## 14. Don't-do list

1. **Don't fill CSVs in F1.5's PR.** This directive ships the templates + tooling; calibration values are Stage B (founder offline work + separate PR).
2. **Don't skip the validation pass.** The converter's validation catches typos, missing rows, out-of-bound values — these are the most common errors when filling 5 CSVs by hand (78 total rows in v3 full scope; 31 in v1 editorial-only Stage B scope; was 162 in v2).
3. **Don't add multi-currency support here.** EUR-only v1 per F1 §5.2. Multi-currency is per-currency calibration directives (F1.5-USD, F1.5-GBP, etc.).
4. **Don't cite external sources as authoritative in `calibration_basis`.** The column is internal documentation for recalibration teams. Per brief v3 §4.2.4, the engine's runtime basis statement to creators uses Frontfiles-positioned language; the calibration_basis is the founder's note for future-self / future-team.
5. **Don't auto-generate "spine" cell values.** Founder picks the anchors. The directive offers spine-cell candidates as workflow guidance, not as required values.
6. **Don't ship the generated SQL outside `_pending/` until F2.** F2 directive timestamps + moves the seed into `supabase/migrations/`.
7. **Don't bypass the floors cross-check.** Even though it's a warning (not an error), every flagged cell deserves explicit founder review before F2.
8. **Don't add per-format exclusivity rows.** Exclusivity is downstream of base price per `EXCLUSIVE_MULTIPLIERS` + brief v3 §4.6 + F1 §3.1. F1.5's CSVs do NOT have an `exclusivity` column.
9. **Don't introduce floors below the platform's commercial reality.** A `min_cents` of €1.00 effectively disables the floor for that combination. Floors should be set at meaningful values (typical: 30-70% of the corresponding small_pub default, depending on format).
10. **Don't merge F2 before this directive ratifies AND the calibration values are filled.** Both gates required per §13.
